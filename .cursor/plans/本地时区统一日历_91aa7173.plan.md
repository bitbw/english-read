# 统一使用用户本地日历日（时区）

## 方案对比：仅请求头 vs 设置页手动时区


| 维度            | A. 仅 `clientFetch` 自动带浏览器时区（`Intl…resolvedOptions().timeZone`） | B. 仅在设置页让用户选手动时区                     |
| ------------- | -------------------------------------------------------------- | ------------------------------------ |
| 上手成本          | 零配置，多数用户与系统日历一致                                                | 需做设置 UI、持久化、迁移；新用户要选一次或给默认值          |
| 准确性           | 与设备/浏览器一致；少数环境（远程桌面、错误系统时区）会偏                                  | 用户显式控制，适合「人在海外仍按国内日界算一天」             |
| 旅行 / 换机       | 自动跟设备变，「今天」自然切换                                                | 不变除非用户改，行为可预期也可「落后」于真实位置             |
| 实现量           | 小：header + 校验 + 各 API 用同一解析函数                                  | 中：DB 字段、PATCH API、设置表单、与 header 的优先级 |
| 服务端 SSR / 邮件等 | 无 DB 时仍依赖请求头或 UTC                                              | 不打开页面也能用库里时区（若以后有服务端定时任务）            |


**结论（推荐）：混合 C** — **默认用浏览器时区（请求头）**；**在设置中增加可选「学习日历时区」**，写入 `users`（或独立偏好表）；服务端统一 `resolveTimeZone(sessionUserId, req)`：**库里有合法覆盖值则用之，否则用 `X-User-Timezone`，再否则 `UTC`**。这样既不强迫所有人进设置，又覆盖手动纠偏与「固定用某国日历」的需求。

若你希望 **第一版最小改动**：可先只做 A（无 DB、无设置页），日后再加 B 的覆盖层而不推翻 API 形状。

## 现状要点

- `[src/lib/reading-time.ts](src/lib/reading-time.ts)`：`utcDayString` / `utcDayKeys` 基于 `toISOString()` 与 `getUTC*`，即 **UTC 自然日**。
- `[src/lib/vocabulary-daily-limit.ts](src/lib/vocabulary-daily-limit.ts)`：日上限 20 用 `vocabularyUtcDayBounds()`，与 UTC 日一致。
- `[src/app/api/reading/time/route.ts](src/app/api/reading/time/route.ts)`：POST 累加的 `day` 字段为 UTC 日；GET 的 `series` 键为 `utcDayKeys`。
- `[src/app/api/review/route.ts](src/app/api/review/route.ts)`：`date=YYYY-MM-DD` 被解析为 `...T00:00:00.000Z`～`23:59:59.999Z`，即 **把该字符串当作 UTC 日**。
- `[src/app/api/review/plan/route.ts](src/app/api/review/plan/route.ts)`：`todayKey` 用 `utcDayString`；按格聚合用 `utcDayString(row.nextReviewAt)`。
- `[src/lib/srs.ts](src/lib/srs.ts)`：`addDays` 使用 `setDate`/`setHours(0,0,0,0)`，在 **Node 默认时区（部署上多为 UTC）** 下推进，并非用户本地午夜语义。
- [src/app/(app)/vocabulary/plan/plan-page-client.tsx](src/app/(app)/vocabulary/plan/plan-page-client.tsx)：初始 `year`/`month` 用 `getUTCFullYear()` / `getUTCMonth()`，与「本地月」不一致。
- [src/components/dashboard/daily-study-chart.tsx](src/components/dashboard/daily-study-chart.tsx)：文案写死「按 UTC 日历」；数据来自 [dashboard/page.tsx](src/app/(app)/dashboard/page.tsx) 服务端用 `utcDayKeys` 查库（SSR 无法拿到浏览器时区）。

## 核心策略

1. **时区传递**：在 `[src/lib/client-fetch.ts](src/lib/client-fetch.ts)` 里为浏览器发起的 `fetch` 自动合并请求头 `X-User-Timezone`，值为 `Intl.DateTimeFormat().resolvedOptions().timeZone`（可做模块级缓存）。凡走 `clientFetch` 的接口（阅读上报、加生词、复习相关等）无需逐处改参数。
2. **服务端解析（混合 C）**：在 `[src/lib/user-timezone.ts](src/lib/user-timezone.ts)` 中实现 `resolveTimeZone(userId, req)`：**若用户已保存 `timeZone`（IANA）且校验通过则优先**；否则读 `x-user-timezone` 并校验；**再否则 `UTC`**。需要 `userId` 的 API 在 `auth()` 后查库一次（可后续加短缓存）；无 session 的路由不适用。
3. **日历与日界**：新增 `[src/lib/user-calendar.ts](src/lib/user-calendar.ts)`（或扩展 `reading-time.ts`），在已校验的 `timeZone` 上提供：
  - `calendarDayKey(timeZone, instant)` → `YYYY-MM-DD`（推荐 `Intl.DateTimeFormat('en-CA', { timeZone, year, month, day })`）
  - `calendarDayKeys(numDays, timeZone)`：从今天（在该时区下的日历日）往前共 `numDays` 天
  - `zonedDayRangeUtc(dayKey, timeZone)`：`[startInclusive, endExclusive)` 两个 `Date`，用于 SQL 里对 `timestamp` 做半开区间，避免 `23:59:59.999` 精度问题
4. **依赖**：项目已有 `date-fns@4`，建议增加 `**@date-fns/tz`**，与 `date-fns` 的 `startOfDay` / `addDays` 等配合 `{ in: tz(timeZone) }` 计算「本地零点 + N 天」的 `Date`，改写 `[src/lib/srs.ts](src/lib/srs.ts)` 中 `getInitialReviewDate`、`calculateNextReview`（及内部 `addDays` 语义），使写入库的 `nextReviewAt` 表示 **用户时区下的日历日凌晨 + 间隔天数** 对应的 UTC 时刻。调用方：`[api/vocabulary/route.ts](src/app/api/vocabulary/route.ts)` POST、`[api/review/submit/route.ts](src/app/api/review/submit/route.ts)` 从 header 取时区传入。

## 数据与兼容性说明

### 是「只改显示」还是「库里的值也会变」？

- **不是纯显示层**：上线后 **新写入** 的规则会变——例如阅读时长按日的 `day` 字符串、生词「今日是否已满 20」所查的 `createdAt` 区间、复习提交后算的下一个 `nextReviewAt`，都按 **解析后的学习时区** 计算。这些是 **以后每次 API 写入时** 的新语义，不是只在前端格式化一下。
- **不会对历史数据做批量 UPDATE**：已有的 `timestamp`（如 `vocabulary.nextReviewAt`、`createdAt`）**保持磁盘上的绝对时刻不变**；没有计划跑迁移去改每一行。因此 **不会出现「全表时间被改掉」** 的情况。

### 存量数据在新方案下可能「看起来」不一致的地方


| 数据                                   | 会不会改库 | 新方案下的影响                                                                                        |
| ------------------------------------ | ----- | ---------------------------------------------------------------------------------------------- |
| `reading_daily_time.day` + `seconds` | 不改旧行  | 旧行的 `day` 是按 **当时 UTC 日** 存的；新行按 **本地日** 存。同一真实日历日在图表上可能拆成两个桶或和本地标签差一天，**仅展示/聚合边界问题**，不是秒数被篡改。 |
| `vocabulary.createdAt`               | 不改    | 日上限 20 只影响 **以后** 的计数区间；不会回溯重算「你去年某天加了几条」。                                                     |
| `vocabulary.nextReviewAt`            | 不改旧值  | 月历/按日筛选用 **新规则** 把同一 UTC 时刻映射到「用户日历日」，与旧代码用 UTC 日切分相比，格子可能略有不同；**到期与否**（和 `now` 比较）仍是同一绝对时刻。   |
| 新词插入、`review/submit`                 | 新写入   | 新的 `nextReviewAt` 按用户时区「本地零点 + N 天」生成，与旧算法写的旧行可能 **相邻几天差一次边界**，属预期差异。                          |


- 未升级的老客户端（不传时区）继续按 **UTC** 解析，行为与现网一致。
